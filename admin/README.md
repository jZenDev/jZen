# admin/ — `@jzen/admin-core`, the react-admin scaffold

`admin/` is the reusable **framework scaffold** for administration:
[`@jzen/admin-core`](package.json).

> Part of the [jZen](https://github.com/jZenDev/jZen) monorepo. Inside the repo an app's panel
> imports it **from source** via a TypeScript/Vite alias (see "The linking mechanism" below),
> versioned in lockstep with the product; this README also stands on its own for a reader who
> meets the package by itself.

It is split like everything else in jZen (ADR-005) — the
scaffold here is schema-generic, and each app assembles it into its own panel under
`apps/<app>/<app>_admin` (today [`apps/zen_demo/zen_demo_admin`](../apps/zen_demo/README.md)),
which registers the domain resources typed off its generated OpenAPI schema. This mirrors
`client/` (framework libs) ↔ `apps/` (app clients) on the TypeScript tier.

Administration is a **react-admin** stack, deliberately — jZen discards a Flutter admin panel
(MANIFESTO). The panel always speaks `X-Zen-Transport: json`; Protobuf binary is for native
apps only.

## What the scaffold provides

All of it is type-generic, bound to no app's schema:

- **`dataProvider.ts`** — a credentialed data provider wired to jZen's pagination convention:
  list endpoints return a **bare JSON array + `Content-Range`** (the stock `ra-data-simple-rest`
  convention), not a wrapper.
- **`authProvider.ts`** — an auth provider backed by the framework's Supabase session cookies.
- **`LoginPage.tsx`** — a login page.
- **`http.ts`** — the shared fetch layer.

## The linking mechanism

A per-app panel imports the scaffold **from source**, via a TypeScript `paths` alias plus a
Vite `resolve.alias` (with React dedupe) — **not** a pnpm dependency edge. This is the
TypeScript analog of the Dart `path:` dep into `client/` and the Maven `<relativePath>`
inheritance: it keeps the repository root language-neutral (no root `pnpm-workspace.yaml`),
gives the app a single copy of React, and needs no build/publish step for the scaffold.

## The types are generated from the contract

The panel consumes the same OpenAPI-documented REST API as every other client, through types
generated by `openapi-typescript`. The generation runs inside the contract-first loop: `task
sync:contracts` builds the backend, emits `openapi.json`, and regenerates the panel's
`src/api/schema.generated.ts` — which is a **tracked** generated file (a frontend developer must
be able to compile the panel without a JDK). Never hand-edit it; fix the `.proto` or the
resource annotation and regenerate. See STANDARDS "Code generation".

## Adding an admin resource

A resource whose data is **identity** (users) is served framework-side by `AdminUserResource`
in `zen-identity`, so every app's panel inherits user administration. A **domain-specific**
resource lands in that app's own server, and the panel registers a react-admin `<Resource>`
against the type generated for it. The list endpoint is bare-array JSON with `Content-Range`;
`Content-Range`/`Accept-Ranges` are in the backend's CORS `exposed-headers` so a cross-origin
panel can read the total.

## Building, testing, running

From the repository root:

```bash
task deps:admin      # install the scaffold and the zen_demo panel (pnpm)
task test:admin      # tsc -b — typechecks the panel and the scaffold it imports from source
task build:admin     # production build of the zen_demo panel
task run:admin       # dev server on http://localhost:5173, proxying /api to the backend
```

`task run:admin` serves the panel only; start the backend it proxies to with `task run:all`
(or the combined `scripts/admin.sh`, which also seeds an admin login — see
[`../scripts/README.md`](../scripts/README.md)). Roles are loaded from the `users` table, never
the JWT, so a freshly registered user must have its `users.role` set to `admin` before it can
sign in to the panel (`scripts/seed-admin.sh`).
